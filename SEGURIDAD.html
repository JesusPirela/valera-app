✅ Cambios ya aplicados Hecho

Estos cambios ya están activos en producción. No tienen que hacer nada.

🔑

Política de contraseñas fuerte

Mínimo 8 caracteres con letras y números obligatorios, y rechazo automático de contraseñas que aparecen en filtraciones de internet (verificación HIBP).

Impacto: aplica solo a contraseñas nuevas — nadie pierde acceso hoy. Ya no se podrán crear cuentas con contraseñas como "123".
🔒

HTTPS forzado + cabeceras de seguridad en la web

Todo el tráfico se redirige a HTTPS y se agregaron cabeceras de protección: nosniff, X-Frame-Options (anti-clickjacking), HSTS, Referrer-Policy y Permissions-Policy.

Impacto: protege contra interceptación de datos y que otros sitios incrusten la app para engañar usuarios. Invisible para los usuarios.
📋

Auditoría de cambios críticos

Nueva tabla audit_log con triggers: cada modificación o borrado de una propiedad o cliente registra quién fue, cuándo, y el estado anterior completo del registro.

Impacto: si alguien borra datos (por error o mala intención) se sabe exactamente quién fue y se puede recuperar la información. Solo admins pueden consultarlo y nadie puede alterarlo desde la app.
🗄️

Revisión completa de Row Level Security (RLS)

Se auditaron las 43 tablas de la base de datos: todas tienen RLS activado con políticas que limitan qué puede ver y modificar cada usuario según su rol.

Impacto: confirmado que un prospectador no puede leer datos de otros ni saltarse permisos consultando la API directamente.
🔍

Revisión de secretos en el código

Se escaneó todo el repositorio de GitHub: no hay llaves privadas ni contraseñas dentro del código fuente.

Impacto: aunque alguien vea el código, no obtiene acceso a la base de datos.
📦

Vulnerabilidad crítica de dependencias corregida

Se corrigió la vulnerabilidad crítica de shell-quote y la de ws con npm audit fix (sin romper nada).

Impacto: quedan 15 vulnerabilidades moderadas en herramientas de build que solo se resuelven actualizando Expo (ver pendiente #10).

🚨 Lo que deben hacer ustedes Esta semana

Acciones que requieren acceso de dueño a las cuentas — yo no las puedo hacer.

1

Cambiar las contraseñas débiles existentes

La política nueva solo aplica a contraseñas nuevas. Las cuentas viejas con contraseñas como "123" siguen entrando.

  1. Entrar a supabase.com/dashboard → proyecto Valera
  2. Authentication → Users
  3. Empezar por las cuentas admin
  4. Pedir a cada usuario que cambie su contraseña desde la app, o usar "Send password recovery" para forzarlo por email
2

Rotar las llaves de Supabase

El service role key y el personal access token han circulado por chats y archivos. Quien los tenga controla toda la base de datos.

  1. Dashboard → Settings → API → "Reset" en service_role key
  2. supabase.com/dashboard/account/tokens → borrar el token viejo "sbp_..." y crear uno nuevo
  3. Actualizar secrets de Edge Functions si alguna usa la llave (Dashboard → Edge Functions → Secrets)
  4. Actualizar secrets de GitHub Actions si aplica (repo → Settings → Secrets → Actions)
El "anon key" NO hay que rotarlo — es público por diseño.
3

Activar doble factor (MFA) para los admins

Ya está habilitado a nivel proyecto; falta que cada admin lo active en su cuenta.

  1. Instalar Google Authenticator (o similar) en el celular de cada admin
  2. Avisarme para desarrollar la pantalla de enrolamiento MFA dentro de la app

⚠️ Siguientes pasos Próximas 2 semanas

Importantes pero no urgentes.

4

Revisar rate limits de login

Limita los intentos de contraseña para frenar ataques de fuerza bruta.

Dashboard → Authentication → Rate Limits → sugerido: 10 intentos por 5 minutos por IP.
5

Verificar backups y probar una restauración

Tener backups que nunca se han probado es como no tener backups.

Dashboard → Database → Backups → confirmar respaldos diarios → hacer una prueba de restauración en un proyecto de prueba (nunca en producción).
6

Correr el Security Advisor de Supabase

Escáner automático incluido en Supabase que detecta configuraciones inseguras.

Dashboard → Advisors → Security → mandarme la lista de advertencias y yo las corrijo.
7

Limitar acceso a Supabase, Hostinger y GitHub

Reducir quién tiene las contraseñas de los paneles a 1-2 personas, y activar 2FA en esas cuentas.

¿Quién tiene hoy acceso al Dashboard de Supabase? ¿Al FTP de Hostinger? ¿Al repo de GitHub? Hacer lista y recortar.

📅 Mediano plazo 1-2 meses

Mejoras de fondo para mantener la seguridad en el tiempo.

8

Actualizar Expo a la versión 56

Resuelve las 15 vulnerabilidades moderadas restantes. Es un cambio grande que necesita pruebas completas — avísenme cuando haya un momento tranquilo sin lanzamientos.

9

Política de altas y bajas de usuarios

Cuando alguien deje la empresa: desactivar su cuenta el mismo día (Dashboard → Authentication → Users → Ban user). Revisar la lista de usuarios activos cada mes.

10

Monitoreo continuo

Revisar Dashboard → Reports una vez por semana (errores de API, picos raros de tráfico). Si quieren, puedo hacer una pantalla de admin para consultar el log de auditoría fácilmente.
